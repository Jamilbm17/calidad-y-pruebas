Seccion 01

Descripcion general

OpenMed Logistics es una plataforma web orientada a la gestion integral de envios y seguimiento logistico. Permite a administradores supervisar operaciones en tiempo real, a clientes registrar y rastrear sus paquetes, y a transportistas actualizar el estado de cada envio a lo largo de su recorrido.

El sistema implementa autenticacion basada en tokens JWT, control de acceso por roles, y expone una API RESTful documentada que sirve de contrato entre el frontend React y el backend FastAPI.

Objetivos del sistema

Rastreabilidad

Registro historico completo de cada envio desde su creacion hasta la entrega final.

Visibilidad operativa

Panel de administracion con KPIs, graficos de actividad y distribuciones de estado.

Control de acceso

Tres roles diferenciados con permisos granulares sobre cada recurso del sistema.

Calidad verificable

Suite de pruebas automatizadas que validan endpoints, logica de negocio y seguridad.

Alcance funcional

El sistema cubre el ciclo de vida completo de un envio: registro del paquete, asignacion de ruta, actualizaciones de estado en transito, y confirmacion de entrega o devolucion. El modulo de reportes genera metricas agregadas para la toma de decisiones operativas.

Arquitectura del sistema

El sistema adopta una arquitectura de tres capas con separacion clara de responsabilidades. La comunicacion entre capas se realiza exclusivamente a traves de HTTP/JSON mediante una API REST.

Capa de Presentacion — Frontend
React 19 + TypeScript 5.8 + Vite 6 + Tailwind CSS 4
puerto :5173
HTTP / JSON — Axios + TanStack Query
Capa de Logica de Negocio — Backend
FastAPI 0.115 + SQLAlchemy 2.0 + Pydantic 2 + JWT (HS256)
puerto :8000
SQLAlchemy ORM — consultas parametrizadas
Capa de Datos — Base de datos
SQLite (desarrollo) / MySQL (produccion)
openmed.db
El frontend y el backend son completamente independientes. El frontend puede desplegarse en cualquier CDN estatico (Vercel, Netlify) mientras el backend corre en un servidor Python separado.

Seccion 04

Modulos del sistema

La aplicacion se organiza en ocho modulos accesibles desde la barra lateral. El acceso a cada modulo depende del rol del usuario autenticado.

Dashboard
KPIs, grafico de actividad diaria y resumen de ultimos envios y seguimientos.
Envios
Registro, busqueda y listado de todos los paquetes con su estado actual.
Clientes
Directorio de usuarios registrados con sus roles y total de envios.
Seguimiento
Registro de cambios de estado con ubicacion, observaciones y hora exacta.
Rutas
Catalogo de rutas logisticas con distancias y tiempos estimados de transito.
Reportes
Graficos de distribucion por estado y peso, metricas de actividad del sistema.
Notificaciones
Centro de alertas sobre cambios de estado y eventos relevantes del sistema.
Configuracion
Gestion del perfil de usuario, preferencias y ajustes de cuenta.
Seccion 05

Roles y permisos

El sistema implementa control de acceso basado en roles (RBAC). El rol se almacena en el payload del token JWT y se verifica en cada endpoint protegido del backend.

◉ Administrador
  • Ver dashboard con KPIs
  • Listar todos los envios
  • Gestionar clientes
  • Registrar seguimientos
  • Acceder a reportes
  • Ver rutas logisticas
◉ Cliente
  • Ver sus propios envios
  • Consultar seguimiento
  • Recibir notificaciones
  • Gestionar su perfil
  • Sin acceso a reportes
  • Sin acceso a rutas
◉ Transportista
  • Ver envios asignados
  • Actualizar estados
  • Registrar ubicacion
  • Gestionar su perfil
  • Sin acceso a clientes
  • Sin acceso a reportes

Seccion 06

Referencia de la API

Todos los endpoints estan disponibles en http://localhost:8000. La documentacion interactiva automatica de FastAPI se accede en /docs (Swagger) y /redoc.

Los endpoints marcados con 🔒 JWT requieren el header Authorization: Bearer <token>.

Autenticacion

POST /auth/register Registro de nuevo usuario

Crea un usuario con hash bcrypt de la contrasena. El rol por defecto es cliente.

{
  "correo": "usuario@ejemplo.com",
  "contrasena": "secret123",
  "nombre_completo": "Juan Perez",
  "telefono": "999888777"
}
POST /auth/login Autenticacion — devuelve token JWT

Recibe credenciales y retorna un access_token JWT con duracion de 24 horas. El payload incluye sub (correo) y rol.

{
  "correo": "usuario@ejemplo.com",
  "contrasena": "secret123"
}
GET /auth/me Perfil del usuario autenticado 🔒 JWT
Retorna los datos del usuario actual extraidos del token JWT validado.

Envios

POST /envios/ Crear nuevo envio 🔒 JWT

Registra un nuevo envio. El codigo de tracking debe ser unico en el sistema.

{
  "codigo_tracking": "TRK-2025-001",
  "descripcion_paquete": "Documentos medicos",
  "peso_kg": 1.5,
  "direccion_origen": "Av. Lima 123, Lima",
  "direccion_destino": "Av. Arequipa 456, Arequipa",
  "id_usuario_cliente": 3
}
GET /envios/{codigo_tracking} Detalle de un envio 🔒 JWT
Retorna el envio identificado por su codigo junto con el ultimo estado de seguimiento registrado.
POST /envios/{codigo_tracking}/seguimiento Agregar actualizacion de estado 🔒 JWT 
{
  "estado": "en_transito",
  "ubicacion_actual": "Lima, San Isidro",
  "observaciones": "Paquete recogido por transportista"
}

Estados validos: registrado en_transito en_deposito entregado devuelto

Admin

GET /admin/stats Estadisticas globales del sistema 🔒 admin
Retorna KPIs: envios del dia, del mes, total, clientes activos, en transito, distribucion por estado y actividad de los ultimos 14 dias.
GET /admin/envios?skip=0&limit=100 Listado paginado de todos los envios 🔒 admin
Incluye el ultimo estado de seguimiento de cada envio calculado via subconsulta.
GET /admin/clientes Directorio de usuarios y total de envios 🔒 admin
Lista todos los usuarios con su conteo de envios asociados.
GET /admin/seguimientos Historial completo de seguimientos 🔒 admin
Retorna todos los registros de seguimiento ordenados por fecha descendente, con el codigo de tracking de su envio.

Seccion 08

Seguridad

Autenticacion JWT

El sistema utiliza JSON Web Tokens firmados con el algoritmo HS256. Los tokens incluyen:

{
  "sub": "usuario@ejemplo.com",   // correo del usuario
  "rol": "admin",                 // rol para RBAC
  "exp": 1751827200               // expiracion en 24 horas
}

Proteccion de contrasenas

Las contrasenas nunca se almacenan en texto plano. Se aplica bcrypt con factor de coste predeterminado (12 rondas). La verificacion usa verify_password(plain, hashed) de la libreria passlib.

Autorizacion por rol

Cada endpoint protegido invoca la dependencia get_current_user que decodifica el token y retorna un diccionario con el correo y el rol. Los endpoints de admin verifican:

if current_user.get("rol") != "admin":
    raise HTTPException(status_code=403, detail="Acceso denegado")

Validacion de datos

Pydantic 2 valida todos los cuerpos de request en el backend antes de procesarlos. En el frontend, Zod valida los formularios antes de enviar cualquier peticion HTTP, eliminando una categoria completa de errores de entrada.

El SECRET_KEY para firmar los tokens JWT debe configurarse como variable de entorno en produccion. No hardcodear en el codigo fuente.
Seccion 09

Guia de instalacion

Requisitos previos

Python 3.11+

Entorno de ejecucion del backend FastAPI.

Node.js 20+ / pnpm

Entorno de ejecucion y gestor de paquetes del frontend.

Git

Para clonar el repositorio del proyecto.

Backend (FastAPI)

1
Clonar el repositorio y entrar al directorio del backend
git clone <url-repositorio>
cd back
2
Crear y activar entorno virtual
python -m venv venv
# Windows
venv\Scripts\activate
# Linux / macOS
source venv/bin/activate
3
Instalar dependencias
pip install -r requirements.txt
4
Ejecutar el servidor de desarrollo
uvicorn main:app --reload --port 8000

El servidor queda disponible en http://localhost:8000. La base de datos SQLite se crea automaticamente.

Frontend (React)

1
Entrar al directorio del frontend
cd front
2
Instalar dependencias con pnpm
pnpm install
3
Iniciar el servidor de desarrollo
pnpm dev

La aplicacion queda disponible en http://localhost:5173.

Generar build de produccion

pnpm build     # genera dist/
pnpm preview   # sirve el build localmente
Seccion 10

Estrategia de pruebas

El proyecto implementa pruebas automatizadas con Pytest en el backend, cubriendo validaciones de entrada, logica de negocio, autenticacion y casos de error.

Pruebas unitarias
Validacion de schemas
Verifican que los modelos Pydantic rechacen datos invalidos (correo mal formado, peso negativo, campos requeridos vacios).
Pruebas de integracion
Flujo de autenticacion
Prueban el ciclo completo: registro → login → token → acceso a recurso protegido con HTTPX sobre DB SQLite en memoria.
Pruebas de negocio
Gestion de envios
Crean, consultan y actualizan envios verificando que el estado se propague correctamente a traves de los seguimientos.
Pruebas de seguridad
Control de acceso
Verifican que los endpoints de admin devuelvan 403 para roles sin privilegios y 401 para tokens invalidos o ausentes.
Pruebas de salud
Health checks
Verifican que el endpoint raiz responda con 200 y confirman que la API este operativa antes de ejecutar suites completas.
Pruebas de duplicados
Restricciones de unicidad
Verifican que el sistema retorne el error correcto al registrar un correo o codigo de tracking ya existente en la base de datos.

Ejecutar la suite de pruebas

# Desde el directorio back/
pytest tests/ -v

# Con reporte de cobertura
pytest tests/ --cov=app --cov-report=term-missing

Estructura de pruebas

back/
  tests/
    test_auth.py          # Registro, login, tokens
    test_envios.py        # CRUD de envios y seguimientos
    test_admin.py         # Endpoints de administrador
    test_validations.py   # Schemas y validaciones Pydantic
    conftest.py           # Fixtures: DB en memoria, usuario de prueba
Las pruebas usan una base de datos SQLite en memoria separada de la base de produccion. Cada test comienza con un estado limpio gracias al fixture db_session con rollback automatico.
